# WIKI.md \u2014 The LLM Wiki Pattern (Part 31 \u2014 Karpathy)\n\n_This workspace follows Andrej Karpathy's three-tier LLM Wiki pattern (April 2026) for organizing agent knowledge._\n\n## The Three Tiers\n\nEvery piece of knowledge lives in exactly one tier. The tiers have different owners, update cadences, and read paths.\n\n### Tier 1: RAW SOURCES (Immutable, agent read-only)\n\n**What:** Canonical ground truth. Code, docs, tickets, transcripts, raw notes. The agent doesn't write here.\n\n**Where in OpenClaw:**\n- `vault/` \u2014 deep storage (projects, decisions, learnings, infrastructure notes)\n- `memory/YYYY-MM-DD.md` \u2014 daily session logs (auto-captured, rolling)\n- `vault/.learnings/HOT.md` \u2014 freshly extracted insights (auto-capture hook)\n\n**Update cadence:** Whenever new stuff happens (new session, user writes a note, auto-capture hook fires).\n\n**Read path:** Agent calls `memory_search()` to find relevant files. Never injected whole into context.\n\n**Key rule:** Append-only. Never delete or rewrite raw sources. Archive instead (move to `vault/03_archive/`).\n\n---\n\n### Tier 2: CURATED SUMMARIES (Model-maintained, hot path)\n\n**What:** Distilled knowledge the agent produced + human refined. \"What we know about X.\" These files are **small, loaded every message**.\n\n**Where in OpenClaw:**\n- `SOUL.md` \u2014 agent identity, tone, core invariants (< 1 KB)\n- `AGENTS.md` \u2014 operational rules, decision tree, security policy (< 2 KB)\n- `MEMORY.md` \u2014 knowledge index + metadata (< 3 KB, pure pointers to vault/)\n- `DREAMS.md` \u2014 readable reflection diary (human-readable consolidation summaries)\n- `skills/*.md` \u2014 named playbooks, wired on demand\n\n**Update cadence:**\n- `SOUL.md`: Rarely (agent personality doesn't change often)\n- `AGENTS.md`: Regularly (new patterns, security rules)\n- `MEMORY.md`: Nightly via memory-core's Deep-phase sweep (dreaming)\n- `DREAMS.md`: Nightly, written by built-in dreaming\n- `skills/*`: On-demand, new skill discovered or old one retired\n\n**Read path:** All of Tier 2 (except `DREAMS.md`) is injected into context on every message. Must stay compact.\n\n**Key rule:** These files are **indexes and rules**, not detail storage. Keep them lean. Push details to `vault/` and link from here.\n\n---\n\n### Tier 3: GENERATED ARTIFACTS (Output, for humans)\n\n**What:** Code, reports, PRs, decisions, documents the agent produces for human consumption.\n\n**Where:** Project-specific (not in workspace/):\n- Generated code → git branches\n- Reports → docs/ or confluence\n- Decisions → ADRs or decision logs\n\n**Update cadence:** Per task.\n\n**Read path:** Humans read this. Agent doesn't loop it back as input (that poisons the ground truth).\n\n**Key rule:** Artifacts never feed back into Tier 1 or 2 without human review.\n\n---\n\n## The Flow\n\n```\nEvery session:\n  SOUL.md + AGENTS.md + MEMORY.md (Tier 2, ~3 KB) injected into context\n    \u2193\n  Agent reads raw sources via memory_search() (Tier 1, on-demand)\n    \u2193\n  Agent produces artifacts (Tier 3, project-specific)\n    \u2193\n  Session ends\n\nNightly:\n  Auto-capture hook extracts insights from session transcript \u2192 vault/.learnings/HOT.md (Tier 1)\n    \u2193\n  Memory-core dreaming processes daily logs (memory/YYYY-MM-DD.md)\n    \u2193\n  Deep-phase sweep: best learnings promoted to MEMORY.md (Tier 2)\n    \u2193\n  DREAMS.md updated with consolidation summary (Tier 2, readable)\n    \u2193\n  Hot learnings auto-archive after 30 days (Tier 1 \u2192 vault/03_archive/)\n```\n\n## Why This Works\n\n**Eliminates tier confusion:**\n- Tier 1 is raw, Tier 2 is curated, Tier 3 is output. No blending.\n- Vault noise doesn't pollute the hot path (Tier 2 stays small).\n- Output doesn't feed back and corrupt input.\n\n**Scales with time:**\n- Vault grows (more raw sources = richer search)\n- Tier 2 stays constant (~8 KB injected per message)\n- Agent gets smarter without paying context cost\n\n**Survives team changes:**\n- Raw sources (Tier 1) are permanent institutional memory\n- Curated summaries (Tier 2) are current best practices\n- Artifacts (Tier 3) are project output\n- Each tier has a clear owner\n\n## What Goes Where: Decision Tree\n\n**Q: I just learned something important.**\n\u2192 Write to `memory/YYYY-MM-DD.md` (today's daily log). Auto-capture hook will extract it to `vault/.learnings/HOT.md` later.\n\n**Q: This learning should be permanent policy.**\n\u2192 After it appears in `MEMORY.md` via dreaming, move it to `AGENTS.md` (if it's a rule) or `SOUL.md` (if it's identity/tone).\n\n**Q: I have architecture/design details that agents need to read.**\n\u2192 Write to `vault/02_reference/` with a claim-based filename (e.g., `why-we-chose-postgres-over-dynamodb.md`). Link from `MEMORY.md`.\n\n**Q: I have a detailed project history/status.**\n\u2192 Write to `vault/projects/project-name.md`. Link from `MEMORY.md`.\n\n**Q: This is a mistake to avoid / lesson learned.**\n\u2192 Write to `vault/learnings/` or `vault/.learnings/HOT.md`. Link from `MEMORY.md` once it's promoted.\n\n**Q: I need to output code/reports/decisions.**\n\u2192 Write to the project repo (git, docs/, etc.). Don't loop back into `vault/` unless the project is *itself* the knowledge base.\n\n---\n\n**See also:** [Part 31 \u2014 The LLM Wiki Pattern In OpenClaw](../openclaw-optimization-guide/part31-the-llm-wiki-pattern-in-openclaw.md)\n
